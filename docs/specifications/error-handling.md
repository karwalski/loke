# Error Handling Framework — Specification

**Epic:** P6 — Error Handling Framework
**Status:** Draft
**Version:** 0.1.0
**Last updated:** 2026-04-05

---

## 1. Overview

loke enforces consistent error handling across three boundaries: the server (Node.js process), the client (browser), and the API communication layer between them. No raw errors, stack traces, or unstructured messages are exposed to users. Every error is captured, correlated, logged, and presented in a controlled manner.

This specification defines:

- Server-side error handling: global catch, correlation IDs, response shape, stack trace suppression (P6.1)
- Client-side error handling: global handlers, toast notifications, rate-limited server reporting (P6.2)
- API client wrapper: network errors, timeouts, HTTP error statuses, JSON parse failures, 5xx retry (P6.3)

### 1.1 Design Alignment

This specification follows the loke design principles:

- **Privacy is the foundation.** Stack traces, internal file paths, and database details are never leaked to the client. Error messages are generic by default; detailed diagnostics exist only in server logs.
- **Warnings must be earned.** Transient network errors are retried silently. Users see error notifications only when something genuinely requires their attention.
- **Do the right thing by default.** Applications get correct error handling by using the platform's API client wrapper and route handlers. Doing the wrong thing (raw `fetch`, unhandled rejections) requires deliberate effort.
- **Auditable.** Every server error is logged with a correlation ID. Users can reference this ID when reporting issues, and developers can trace the full error chain in logs.

## 2. Server Error Handling (P6.1)

### 2.1 Global Error Catch

The server installs two global error handlers:

1. **Middleware error boundary.** The error handling middleware (pipeline stage 11 in the HTTP server spec) catches any error thrown by route handlers or earlier middleware. This is the primary catch point for request-scoped errors.

2. **Process-level handlers.** For errors that escape the middleware pipeline:
   - `process.on('uncaughtException', handler)` — logs the error and initiates graceful shutdown. An uncaught exception indicates a programming error; the process is in an unknown state and must restart.
   - `process.on('unhandledRejection', handler)` — logs the error with a warning. Does not crash the process (Node.js default behaviour since v15), but logs at `error` level with full context.

### 2.2 Error Response Shape

Every error response from the server uses the following shape. This is the canonical error format for the entire platform — all middleware, route handlers, and platform endpoints use it.

```typescript
interface ErrorResponse {
  error: {
    code: string;          // machine-readable error code from the registry (Section 2.4)
    message: string;       // human-readable message, safe to display to users
    requestId: string;     // correlation ID (matches X-Request-Id response header)
    details?: unknown[];   // structured details (validation errors, etc.) — never stack traces
  };
}
```

**Rules:**

1. The `code` field is always a string from the error code registry (Section 2.4). It is stable across versions and safe for client-side programmatic matching.
2. The `message` field is a user-safe description. It may be an i18n key (e.g. `loke.error.rate_limit_exceeded`) or an English fallback string. It never contains internal details.
3. The `requestId` is the UUID v4 generated by the request ID middleware. It appears in both the response body and the `X-Request-Id` response header.
4. The `details` array is present only when additional structured information is available (e.g. validation errors). It never contains stack traces, file paths, SQL queries, or internal state.

### 2.3 Correlation ID Flow

Every request receives a unique correlation ID at the start of the middleware pipeline:

```
Client request
  → Request ID middleware generates UUID v4 → attaches to RequestContext
    → All log entries for this request include the ID
    → Error response includes the ID in `error.requestId`
    → Response header `X-Request-Id` carries the ID back to the client
```

The client stores this ID and includes it in error reports (Section 3), enabling end-to-end tracing from client error through server logs.

### 2.4 Error Code Registry

Error codes are string constants, namespaced by category. Platform codes use the `loke.` prefix. Applications define their own prefix.

#### Platform Error Codes

| Code | HTTP Status | Description |
|------|-------------|-------------|
| `VALIDATION_ERROR` | 400 | Request validation failed. `details` contains `ValidationError[]`. |
| `MALFORMED_JSON` | 400 | Request body is not valid JSON. |
| `UNAUTHORIZED` | 401 | Authentication required but not provided or invalid. |
| `FORBIDDEN` | 403 | Authenticated but not authorised for this resource. |
| `NOT_FOUND` | 404 | Route or resource not found. |
| `METHOD_NOT_ALLOWED` | 405 | HTTP method not supported for this route. |
| `CONFLICT` | 409 | Resource state conflict (e.g. duplicate creation). |
| `PAYLOAD_TOO_LARGE` | 413 | Request body exceeds `maxBodySize`. |
| `UNSUPPORTED_MEDIA_TYPE` | 415 | Request content type is not `application/json`. |
| `RATE_LIMIT_EXCEEDED` | 429 | Rate limit exceeded. `Retry-After` header is set. |
| `REQUEST_TIMEOUT` | 408 | Request exceeded the configured timeout. |
| `INTERNAL_ERROR` | 500 | Unhandled server error. Details are logged server-side; the client receives a generic message. |
| `SERVICE_UNAVAILABLE` | 503 | A required subsystem (database, integration) is unhealthy. |

#### Application Error Code Convention

Applications define error codes under their own namespace:

```typescript
// Example: calendar application
const CALENDAR_ERRORS = {
  EVENT_OVERLAP: 'calendar.event_overlap',       // 409
  PROVIDER_SYNC_FAILED: 'calendar.sync_failed',  // 502
  QUOTA_EXCEEDED: 'calendar.quota_exceeded',      // 429
} as const;
```

### 2.5 Stack Trace Handling

Stack traces are **never** included in HTTP responses, regardless of environment. In development mode, stack traces are logged at `debug` level in the server logs. In production mode, stack traces are logged at `error` level but are still never sent to the client.

The error handling middleware maps any thrown error to the appropriate error code and a generic message:

```typescript
// What the handler throws
throw new Error('SQLITE_BUSY: database is locked');

// What the client receives
{
  "error": {
    "code": "INTERNAL_ERROR",
    "message": "An unexpected error occurred. Please try again.",
    "requestId": "req_a1b2c3d4"
  }
}

// What the server logs
{
  "level": "error",
  "requestId": "req_a1b2c3d4",
  "error": "SQLITE_BUSY: database is locked",
  "stack": "Error: SQLITE_BUSY...\n    at Database.exec (...)",
  "method": "POST",
  "path": "/api/v1/calendar/events"
}
```

### 2.6 Typed Application Errors

The platform provides an `AppError` base class that applications use to throw errors with explicit codes and HTTP statuses:

```typescript
class AppError extends Error {
  constructor(
    public readonly code: string,
    public readonly statusCode: number,
    message: string,
    public readonly details?: unknown[],
  ) {
    super(message);
    this.name = 'AppError';
  }
}

// Convenience subclasses
class NotFoundError extends AppError {
  constructor(message = 'Resource not found') {
    super('NOT_FOUND', 404, message);
  }
}

class ValidationError extends AppError {
  constructor(message: string, details: unknown[]) {
    super('VALIDATION_ERROR', 400, message, details);
  }
}

class ConflictError extends AppError {
  constructor(message = 'Resource conflict') {
    super('CONFLICT', 409, message);
  }
}
```

The error handling middleware recognises `AppError` instances and uses their `code`, `statusCode`, and `details` directly. Any other thrown value is treated as an `INTERNAL_ERROR` (500).

## 3. Client Error Handling (P6.2)

### 3.1 Global Handlers

The client installs two global error handlers on page load:

```typescript
// Uncaught exceptions
window.addEventListener('error', (event) => {
  clientErrorHandler.capture(event.error, { source: 'uncaught' });
});

// Unhandled promise rejections
window.addEventListener('unhandledrejection', (event) => {
  clientErrorHandler.capture(event.reason, { source: 'unhandled_rejection' });
});
```

These handlers ensure that no error silently disappears. Every captured error follows the same pipeline: classify, notify user, report to server.

### 3.2 Error Classification

Captured errors are classified to determine how they should be presented:

| Classification | Examples | User notification | Report to server |
|---------------|----------|-------------------|------------------|
| **Network error** | Fetch failed, connection refused, DNS failure | Toast: "Unable to connect to the server. Check that loke is running." | Yes (when connectivity restores) |
| **Server error (5xx)** | 500, 502, 503 responses | Toast: "Something went wrong. The error has been logged (ID: {requestId})." | No (already logged server-side) |
| **Client logic error** | TypeError, ReferenceError, null dereference | Toast: "Something went wrong. The error has been reported." | Yes |
| **Render error** | Failed to mount component, template error | Toast: "A display error occurred. Try refreshing the page." | Yes |
| **Timeout** | API request exceeded client-side timeout | Toast: "The request timed out. Please try again." | Yes |

### 3.3 Toast Notifications

Errors are presented to users as toast notifications (P3.8), never as raw error messages, console output, or browser alert dialogs.

**Toast properties for error notifications:**

| Property | Value |
|----------|-------|
| Type | `error` |
| Duration | `8000ms` (auto-dismiss; user can dismiss earlier) |
| Position | Top-right, below the header |
| Content | Classification-specific message (see table above). Includes `requestId` when available. |
| Action | Optional "Details" link that expands to show the error code and request ID for support purposes. |
| Screen reader | Announced via `aria-live="assertive"`. |

**What users never see:**

- Stack traces
- Raw error messages (`Cannot read properties of undefined`)
- HTTP status codes as the primary message
- JSON error objects

### 3.4 Rate-Limited Server Reporting

Client errors are reported to the server for centralised logging via a dedicated endpoint:

```
POST /api/v1/errors/report
```

**Report payload:**

```typescript
interface ClientErrorReport {
  timestamp: string;               // ISO 8601
  classification: string;          // 'network' | 'server' | 'client_logic' | 'render' | 'timeout'
  message: string;                 // error message (sanitised — no user data)
  stack?: string;                  // stack trace (development only; stripped in production builds)
  source: string;                  // 'uncaught' | 'unhandled_rejection' | 'api_client' | 'component'
  url: string;                     // page URL (hash route, no query params)
  userAgent: string;               // navigator.userAgent
  requestId?: string;              // correlation ID if the error relates to an API call
}
```

**Rate limiting:**

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Max reports per minute | 5 | Prevents error floods from tight loops or rapid re-renders. |
| Deduplication window | 60 seconds | Identical errors (same message + source) within this window are counted but not re-sent. |
| Batch interval | 5 seconds | Reports are batched and sent together to reduce request volume. |
| Queue size | 50 | Maximum queued reports. Oldest are dropped when the queue is full. |

When the rate limit is reached, subsequent errors within the window are silently dropped from the reporting queue. They are still counted locally and the count is included in the next report batch.

**Server-side handling:**

The `/api/v1/errors/report` endpoint:

1. Validates the payload against the schema.
2. Logs each report at `warn` level with the client-provided `requestId` (if present) for cross-referencing.
3. Never stores raw error reports in the database (they may contain transient state). Logging only.
4. Rate-limited server-side at 10 reports per minute per client to prevent abuse.

### 3.5 Development Mode

In development mode (`NODE_ENV=development`):

- Stack traces are included in client error reports.
- An additional console error is logged with full details.
- Toast notifications include a "Copy debug info" button that copies the full error (code, message, stack, request ID) to the clipboard.

In production builds:

- Stack traces are stripped from reports.
- Console logging of errors is suppressed (all reporting goes through the toast + server pipeline).

## 4. API Client Wrapper (P6.3)

### 4.1 Overview

The platform provides an API client wrapper that applications use instead of raw `fetch()`. The wrapper handles network errors, timeouts, HTTP error statuses, JSON parse failures, and retries — so that application code only deals with successful responses or well-typed errors.

```typescript
import { api } from '@loke/client';

// Instead of raw fetch:
const events = await api.get('/api/v1/calendar/events', {
  query: { from: '2026-04-01', to: '2026-04-30' },
});

// events is typed as the parsed JSON response
// All error cases are handled by the wrapper
```

### 4.2 API Client Interface

```typescript
interface ApiClient {
  get<T>(path: string, options?: RequestOptions): Promise<T>;
  post<T>(path: string, body: unknown, options?: RequestOptions): Promise<T>;
  put<T>(path: string, body: unknown, options?: RequestOptions): Promise<T>;
  patch<T>(path: string, body: unknown, options?: RequestOptions): Promise<T>;
  delete<T>(path: string, options?: RequestOptions): Promise<T>;
}

interface RequestOptions {
  query?: Record<string, string | number | boolean>;
  headers?: Record<string, string>;
  timeout?: number;              // override default timeout (ms)
  retry?: RetryConfig | false;   // override default retry; false disables retry
  signal?: AbortSignal;          // external abort signal
}

interface RetryConfig {
  maxAttempts: number;           // total attempts including the first (default: 3)
  baseDelay: number;             // initial delay in ms (default: 1000)
  maxDelay: number;              // maximum delay in ms (default: 10000)
  retryOn?: number[];            // HTTP statuses to retry (default: [500, 502, 503, 504])
}
```

### 4.3 Error Handling Pipeline

Every request passes through the following error handling pipeline:

```
api.get('/path')
  │
  ├─ 1. Build request (URL, headers, body serialisation)
  │
  ├─ 2. Send request via fetch()
  │     │
  │     ├─ Network error? → classify as NetworkError
  │     ├─ Timeout?       → classify as TimeoutError
  │     └─ Response received
  │
  ├─ 3. Check HTTP status
  │     │
  │     ├─ 2xx?   → parse JSON → return data
  │     ├─ 4xx?   → parse error body → throw ApiError (no retry)
  │     ├─ 5xx?   → retry logic (see Section 4.4)
  │     └─ Other? → throw ApiError
  │
  ├─ 4. Parse JSON response
  │     │
  │     ├─ Valid JSON?   → return parsed data
  │     └─ Invalid JSON? → throw ParseError
  │
  └─ 5. Return typed result or throw typed error
```

### 4.4 Retry Strategy

The API client retries failed requests under specific conditions:

**Retryable conditions:**

| Condition | Retried | Rationale |
|-----------|---------|-----------|
| HTTP 500 (Internal Server Error) | Yes | Transient server errors may resolve on retry. |
| HTTP 502 (Bad Gateway) | Yes | Upstream service may recover. |
| HTTP 503 (Service Unavailable) | Yes | Service may be restarting. |
| HTTP 504 (Gateway Timeout) | Yes | Upstream timeout may not recur. |
| Network error (fetch failed) | Yes | Transient connectivity loss. |
| HTTP 4xx | No | Client errors are deterministic; retrying will not help. |
| HTTP 429 (Too Many Requests) | Special | Retried after `Retry-After` delay, if header is present. Does not count against `maxAttempts`. |
| JSON parse failure | No | Response is malformed; retrying is unlikely to help. |
| Timeout (client-side) | No | The server may still be processing; retrying could cause duplicate work. |

**Backoff strategy:**

The client uses exponential backoff with jitter:

```
delay = min(baseDelay * 2^(attempt - 1) + random(0, baseDelay), maxDelay)
```

| Attempt | Base delay | Calculated delay (no jitter) | With jitter (range) |
|---------|------------|------------------------------|---------------------|
| 1 | 1000ms | 1000ms | 1000-2000ms |
| 2 | 1000ms | 2000ms | 2000-3000ms |
| 3 | 1000ms | 4000ms | 4000-5000ms |

**Default retry configuration:**

```typescript
const DEFAULT_RETRY: RetryConfig = {
  maxAttempts: 3,
  baseDelay: 1000,
  maxDelay: 10000,
  retryOn: [500, 502, 503, 504],
};
```

**Retry logging:**

Each retry attempt is logged at `debug` level:

```
[debug] API retry: GET /api/v1/health — attempt 2/3, status 503, next retry in 2340ms (requestId: req_a1b2c3d4)
```

### 4.5 Typed Error Classes

The API client throws typed errors that applications can catch and handle:

```typescript
class ApiError extends Error {
  constructor(
    public readonly code: string,         // error code from the server response
    public readonly statusCode: number,   // HTTP status
    message: string,
    public readonly requestId: string,    // correlation ID
    public readonly details?: unknown[],  // structured details from the server
  ) {
    super(message);
    this.name = 'ApiError';
  }
}

class NetworkError extends Error {
  constructor(
    message: string,
    public readonly cause?: Error,        // original fetch error
  ) {
    super(message);
    this.name = 'NetworkError';
  }
}

class TimeoutError extends Error {
  constructor(
    public readonly timeout: number,      // configured timeout in ms
    public readonly path: string,         // request path
  ) {
    super(`Request to ${path} timed out after ${timeout}ms`);
    this.name = 'TimeoutError';
  }
}

class ParseError extends Error {
  constructor(
    public readonly statusCode: number,   // HTTP status of the unparseable response
    public readonly requestId: string,    // correlation ID
    public readonly rawBody: string,      // first 200 characters of the response body
  ) {
    super('Failed to parse server response as JSON');
    this.name = 'ParseError';
  }
}
```

**Usage in application code:**

```typescript
import { api, ApiError, NetworkError, TimeoutError } from '@loke/client';

try {
  const events = await api.get<Event[]>('/api/v1/calendar/events');
  renderEvents(events.data);
} catch (err) {
  if (err instanceof ApiError && err.statusCode === 404) {
    showEmptyState();
  } else if (err instanceof NetworkError) {
    showOfflineIndicator();
  } else if (err instanceof TimeoutError) {
    showTimeoutMessage();
  } else {
    // All other errors are already handled by the global handler (Section 3.1)
    // Re-throw to let the global handler capture it
    throw err;
  }
}
```

### 4.6 Request ID Forwarding

The API client reads the `X-Request-Id` header from every server response and stores it. When an error occurs, the request ID is included in the thrown error and in any client error report sent to the server. This enables end-to-end tracing:

```
Client click → api.post() → server logs (requestId: req_abc) → error thrown →
  → toast shows "Error (ID: req_abc)" → user includes ID in support request →
  → developer searches logs for req_abc → full request trace
```

### 4.7 Default Timeout

The API client has a default timeout of 30 seconds, matching the server-side `requestTimeout` default. Individual requests can override this:

```typescript
// Long-running operation with extended timeout
const result = await api.post('/api/v1/pipeline/run', payload, {
  timeout: 120000,  // 2 minutes
  retry: false,     // don't retry long operations
});
```

### 4.8 Integration with Client Error Handling

The API client integrates with the client error handler (Section 3):

1. **Automatic reporting.** When the API client catches an error that it cannot retry, it reports the error to the client error handler before throwing. The global handler shows the toast and queues the server report.
2. **No double-reporting.** If the application catches the error and handles it (e.g. showing an empty state for a 404), the error is not reported again. The client error handler only fires for errors that propagate to the global handlers.
3. **Network error queuing.** When a `NetworkError` occurs, the error report is queued and sent when connectivity restores (via the batch interval).

## 5. Error Handling Flow (End-to-End)

The following shows the complete error flow for a server error, from origin to user notification:

```
1. Route handler throws an error
   │
2. Error handling middleware catches it
   ├─ Is it an AppError?
   │   ├─ Yes → use its code, statusCode, message, details
   │   └─ No  → map to INTERNAL_ERROR (500), generic message
   │
3. Log the error server-side
   ├─ Level: error
   ├─ Fields: requestId, error message, stack (server logs only), method, path
   │
4. Send error response to client
   ├─ HTTP status from the error code
   ├─ Body: { error: { code, message, requestId } }
   ├─ Header: X-Request-Id
   │
5. API client receives the response
   ├─ Status 5xx? → retry (up to maxAttempts)
   │   ├─ All retries failed → throw ApiError
   │   └─ Retry succeeds → return data
   ├─ Status 4xx? → throw ApiError (no retry)
   │
6. Application code receives the error
   ├─ Catches and handles specifically? → done (no toast)
   └─ Does not catch? → propagates to global handler
       │
7. Global handler captures the error
   ├─ Show toast notification to user
   ├─ Queue error report for server (rate-limited)
   └─ Log to console (development only)
```

## 6. Relationship to Other Specifications

| Specification | Relationship |
|---------------|-------------|
| **Platform HTTP Server (P1)** | The error response shape defined here is used by the server's error handling middleware (P1 pipeline stage 11). Validation errors (P1.4) use the `ValidationError` structure. Rate limit errors (P1.5) use the `RATE_LIMIT_EXCEEDED` code. |
| **Notification System (P3.8)** | Client error toasts are rendered through the platform notification system. Error notifications use the `error` variant. |
| **Integration Framework (P5)** | The API client wrapper (Section 4) shares retry and backoff patterns with the base HTTP client (P5.3). The API client is for internal client-to-server communication; P5.3 is for server-to-external-service communication. |
| **Structured Logging (F1.3)** | All server error logs use the structured logger with correlation IDs. The `requestId` field in error responses matches the correlation ID in log entries. |
| **Health Checks (F1.7)** | When a health check probe fails, the server returns `SERVICE_UNAVAILABLE` (503) using the error response shape defined here. |

## 7. Testing Considerations

| Area | Test approach |
|------|---------------|
| Error response shape | Unit test: throw various error types from a handler, verify response matches `ErrorResponse` schema exactly. |
| Stack trace suppression | Integration test: trigger a server error, verify the response body contains no stack trace. Verify the server log does contain it. |
| Correlation ID flow | Integration test: make a request, verify `X-Request-Id` header matches `error.requestId` in the response body and appears in server logs. |
| Client error reporting | Unit test: simulate uncaught error, verify toast is shown, verify report is queued. Verify rate limiting drops excess reports. |
| API client retry | Unit test: mock server returning 503 three times then 200, verify three retries then success. Mock server returning 400, verify no retry. |
| API client timeout | Unit test: mock server with delayed response, verify `TimeoutError` is thrown after configured timeout. |
| API client JSON parse | Unit test: mock server returning invalid JSON with 200 status, verify `ParseError` is thrown. |
| Rate limiting (client reports) | Unit test: fire 10 errors in 1 second, verify only 5 reports are queued. |
| Typed error classes | Unit test: throw each error class, verify `instanceof` checks work and properties are correct. |
